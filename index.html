Workflow

从内容抓取到最终发布，一条链路完成闭环

这个页面重点突出“流程收敛”而不是“功能堆砌”，因为真正让工具显得高级的，是清晰而可靠的系统感。

Step 1

采集与整理

从目标内容中提取正文与媒体，快速转成可处理的稿件资产。

Step 2

本地落盘

把媒体保存到本地目录，并为后续修改、归档和重试保留稳定输入。

Step 3

同步发布

按目标账号执行发布编排，保持多个子工作区之间的动作一致性。

Step 4

观测与重试

通过状态查询、等待、取消和重试，把不可控的人工流程变成可处理任务。

Interfaces

桌面负责体验，CLI 负责规模化

对个人操作者来说，图形界面足够直接；对脚本、自动化或 agent 来说，CLI 让能力可以被程序化调用。

桌面工作台

适合处理账号管理、子工作区打开、内容检查、人工确认与即时观察。

账号层 在侧边栏添加、编辑、删除账号，并按账号拉起独立子工作区。
稿件层 提取正文、调用内置多种风格的图文模板、下载媒体、复制内容到剪贴板，减少重复切换页面。
发布层 执行单篇或批量同步发布，并在同一界面里看到任务进度与状态变化。

好的桌面工具不只是把按钮摆上去，而是让复杂流程在视觉上也能被用户一眼读懂。

XHS Studio 的首页应该传递这种控制感。

CLI / Agent 工作流

适合把固定动作接入脚本、自动化调度、预检查与批量操作。

预检查 用 `publish preflight` 提前确认媒体、目标账号和执行前置条件是否齐备。
执行 用 `publish run` 发起任务，用 `publish wait` 等待结果，必要时 `retry` 或 `cancel`。
运行时 GUI 与 CLI 共用同一套业务运行时，避免出现两套逻辑分叉和能力不一致。

Publish Protocol shared runtime 

xhs publish preflight --draft draft_042
xhs publish run --file ./tmp/publish.json
xhs publish status --job job_042
xhs publish retry --job job_042
